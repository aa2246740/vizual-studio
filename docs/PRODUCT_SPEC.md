# Vizual Studio 产品文档

状态：内部种子版产品规格  
默认语言：中文  
当前主线：商业级 HTML PPT  
底层依赖：Vizual Core

## 1. 产品愿景

Vizual Studio 是一个面向 AI Agent 时代的专业内容协作平台。它的核心目标不是替代 PowerPoint，也不是做一个泛化设计工具，而是让 AI Agent 生成的分析、报告、图表和业务判断变成真正可阅读、可互动、可批注、可修订、可交付的视觉作品。

第一阶段只打深一条路线：

> 让用户和 AI 一起制作商业级 PPT。源格式是 HTML，图表和数据展示由 Vizual 渲染，协作方式是对话、批注、直接编辑和 liveControl 微调。

长期目标是：

> 成为 AI Agent 的视觉工作台和展示运行时，让任何接入 Vizual 的 Agent 都具备专业可视化、交互式报告和协作交付能力。

## 2. 第一性原理

### 2.1 AI 输出不能只停留在文字

很多 Agent 当前只能输出纯文字或 Markdown。对于数据分析、经营汇报、管理驾驶舱、风险监控、策略复盘这类任务，纯文字会让用户难以判断趋势、结构、异常和优先级。

Vizual Studio 要解决的是：

- 让 Agent 的结果天然图文并茂。
- 让数据、图表、结论和行动建议放在同一个可交互画布里。
- 让用户能继续追问、批注、修改和导出，而不是只收到一段文本。

### 2.2 Vizual 是运行时，Studio 是产品

Vizual Core 负责底层能力：

- 结构化可视化规格。
- 图表、表格、KPI、布局、DocView 等组件。
- 设计风格标准解析与主题映射。
- liveControl。
- 批注、Agent bridge、导出等原语。

Vizual Studio 负责产品体验：

- PPT 项目。
- 幻灯片画布。
- Agent 协作。
- 批注和修订循环。
- 设计风格控制。
- 导出交付。
- 未来的项目、历史、权限、分享和团队协作。

### 2.3 HTML 是源格式

Studio 第一阶段的 PPT 是 HTML-first。原因：

- HTML 更适合 Agent 生成、修改和验证。
- HTML 能承载动态可视化、交互、动画、批注和版本信息。
- HTML 能导出 PDF、PNG、未来也可以适配 PPTX。
- 不应该让 PPTX 的格式限制反过来约束表现力。

### 2.4 用户改内容，Agent 改结构和风格

用户应该能直接在画布上改字，不需要每个文字修改都通过 Agent。

更适合 Agent 的事情是：

- 改图表类型。
- 筛选数据。
- 改布局。
- 强化结论。
- 改设计风格。
- 处理批注。
- 生成修订建议。

### 2.5 所有 Agent 修改都必须可审计

Agent 不应该悄悄覆盖用户内容。成熟工作流必须有：

- 批注对象。
- 修订记录。
- 接受/拒绝。
- 版本历史。
- 可追溯的源数据和上下文。

## 3. 产品定位

### 3.1 当前定位

Vizual Studio 是一个面向内部种子用户的 AI 协作式商业 PPT 平台。

它适合：

- 数据分析 Agent 的结果展示。
- 经营分析汇报。
- 管理层报告。
- 周报、月报、专题分析。
- 数据产品里的可视化回答。
- AI Agent 平台里的高级可视化输出页。

### 3.2 暂不定位为

当前不做：

- 通用 PowerPoint 克隆。
- Figma 式自由设计工具。
- 面向大众的模板网站。
- 纯低代码 BI 工具。
- 完整多人在线协作平台。
- 专业视频制作工具。

这些方向未来可以衍生，但第一阶段不作为核心目标。

## 4. 核心用户

### 4.1 业务用户

业务用户关心的是结果是否清楚、可信、能交付。

典型需求：

- “帮我把这份经营数据做成汇报。”
- “这页结论太弱，写得更像给管理层看的。”
- “这张图看不出拐点，换一种表达。”
- “只看华东区，导出给领导。”
- “这页我批注了，帮我改。”

### 4.2 数据分析 Agent

数据分析 Agent 负责取数、分析和生成内容。

它需要 Studio 提供：

- 可渲染的 deck artifact。
- 图表和表格组件。
- 批注目标上下文。
- 修订写回能力。
- 导出能力。

### 4.3 Agent 开发者

Agent 开发者希望快速接入，不想重新造图表和报告 UI。

他们需要：

- 清晰的 SDK。
- 稳定的 artifact schema。
- 可测试的 host bridge。
- 冷启动 Agent 可读懂的技能和文档。
- 明确的错误提示和调试工具。

### 4.4 平台前端开发者

平台开发者负责把 Studio 或 Vizual 嵌入到企业系统、SaaS 后台、Chatbot 小窗或完整对话页中。

他们需要：

- 可嵌入组件。
- 独立页面模式。
- 事件和 action bridge。
- 样式隔离。
- 导出接口。
- 权限和项目状态接口。

### 4.5 审阅者和管理者

审阅者不关心技术，只关心内容是否准确、表达是否专业。

他们需要：

- 批注。
- 版本。
- 接受/拒绝修订。
- 导出。
- 历史可追溯。

## 5. 核心价值

### 5.1 对用户

- 把 AI 回复从“文字”提升为“可交付作品”。
- 让复杂数据变成直观图表和报告。
- 能像审稿一样对 AI 输出批注和修订。
- 能导出 HTML/PDF，服务真实业务场景。

### 5.2 对 Agent

- Agent 不需要自己写前端。
- Agent 只要输出结构化 artifact，即可拥有专业渲染能力。
- Agent 可以读取批注上下文并做精准修改。
- Agent 可以通过 liveControl 暴露安全、可控的用户调节能力。

### 5.3 对 Vizual 生态

- Studio 是 Vizual Core 的最佳实践应用。
- Studio 反向验证 Core 缺什么。
- Studio 是展示 Vizual 价值的完整产品样板。

## 6. 成功标准

第一版成功标准：

- 打开页面后，用户能立刻理解这是一个 AI 协作做 PPT 的产品。
- 左侧是内容和画布，右侧是 Agent 协作，而不是功能堆砌。
- 用户能直接编辑标题和正文。
- 用户能选中对象并批注。
- 用户能通过 liveControl 微调图表和设计风格。
- Agent 能通过浏览器桥读取批注并写回修订。
- Vizual 图表、KPI、表格能作为 slide 内容出现。
- 能导出 PDF/HTML 初稿。

下一阶段成功标准：

- Agent 能从零生成完整 deck。
- 批注能精确绑定文本、图表、数据点和页面区域。
- 修订有 diff、预览、接受/拒绝。
- 设计风格覆盖所有 Vizual 组件。
- 历史对话中的旧 artifact 能重新打开并继续修改。
- 多个可视化块的 liveControl 状态互不污染。

## 7. 产品结构

Vizual Studio 的完整产品由六层组成：

1. 项目层：项目、历史、数据源、设计风格、导出记录。
2. Deck 层：幻灯片列表、页面结构、布局、讲稿备注。
3. Canvas 层：16:9 HTML 画布、文本、图表、表格、组件块。
4. Agent 协作层：对话、批注、修订、版本。
5. liveControl 层：用户可理解的参数调节器。
6. Export 层：PDF、HTML、图片、未来 PPTX。

## 8. 页面与功能设计

## 8.1 工作台首页

当前状态：规划中。  
第一版入口暂时直接进入编辑器。

### 页面目标

让用户管理 PPT 项目，并快速开始一个新的 AI 协作任务。

### 核心区域

- 最近项目。
- 新建项目入口。
- 设计风格入口。
- 示例项目。
- 导入历史 artifact。

### 关键操作

- 新建 PPT。
- 从数据分析任务生成。
- 从已有报告继续编辑。
- 打开最近项目。
- 导入 HTML deck。

### Agent 能力

- 根据用户一句话创建项目。
- 识别项目类型：经营分析、专题报告、周报、竞品分析、董事会汇报。
- 推荐页面大纲。

### 成功标准

- 用户不需要理解 Vizual，也能开始制作一份 PPT。
- 首页不暴露内部 schema 和协议名。

## 8.2 PPT 编辑器

当前状态：已做第一版。

### 页面目标

这是 Studio 的核心页面。它要支持用户和 Agent 一起完成 PPT 的生成、编辑、批注、修订和导出。

### 页面布局

采用三栏编辑器：

- 顶部：项目栏和导出入口。
- 左侧：幻灯片缩略图列表。
- 中间：当前幻灯片画布。
- 右侧：Agent 协作面板。

### 顶部栏

功能：

- 显示产品名。
- 显示当前项目名。
- 显示内部预览状态。
- 导出 PDF。
- 导出 HTML。

设计原则：

- 不展示“用户旅程”。
- 不把内部协议名作为主视觉。
- 顶部只服务当前编辑任务。

### 左侧幻灯片列表

功能：

- 展示所有幻灯片。
- 展示页码。
- 展示标题。
- 展示状态：草稿、待审、已确认。
- 切换当前页。

后续增强：

- 拖拽排序。
- 新增/复制/删除页面。
- 页面分组。
- 页面状态批量管理。
- 缩略图真实预览。

### 中央画布

功能：

- 16:9 HTML slide。
- 标题可直接编辑。
- 正文可直接编辑。
- 图表区域嵌入 VizualRenderer。
- 页脚显示品牌或项目标识。
- 备注区支持讲稿备注。

当前可选对象：

- 整页。
- 标题。
- 正文。
- 图表。

后续可选对象：

- 文本选区。
- 图表数据点。
- 表格单元格。
- 图片。
- KPI 卡片。
- 自由布局块。
- 页面背景。

### 右侧 Agent 协作面板

右侧是协作中心和对象属性面板，不再用顶部的“整页/标题/正文/图表/图片”行让用户预选对象。用户直接点击画布对象，右侧自动切换上下文。它包含三个模式：

1. AI 修改：把当前对象的修改要求提交给 Agent。
2. 手动微调：对当前对象做确定性样式调整。
3. 修订记录：查看、接受、拒绝 Agent 产生的修订。

顶部工具模式：

- 选择：默认浏览和选中对象。
- Edit：文字对象可直接在画布上像 Word 一样编辑。
- Draw：用手绘区域标注告诉 Agent 要改哪里。
- Click：点选具体对象并把说明加入标注队列。

## 8.3 Agent 协作流

当前状态：已做第一版。

### 页面目标

让用户用自然语言要求 Agent 修改当前 deck，并且每条请求都带上当前选中对象。这里的“批注”不是普通评论，而是提交给 Agent 的目标化修改任务。

### 用户可以说

- “这页改成折线图。”
- “结论写得更具体一点。”
- “把这页变成明细表。”
- “只看华东区。”
- “这页更适合管理层，少一点数据，多一点判断。”
- “这个标题不要换行。”
- “这张图改成组合图并强调拐点。”

### 当前第一版能力

- 用户直接点击对象：整页、标题、正文、图表、图片。
- 用户在协作输入框描述修改要求。
- 系统把请求记录为 open task，并显示在协作流顶部。
- 页面触发 `vizual-studio:comment-added` 事件，供内部 Agent 监听。
- “处理待办”可以模拟 Agent 消费这些请求并生成修订。
- Draw/Click 模式支持底部标注输入条和 Queue，一次提交多条带 target/bbox 的修改任务。

### 后续正式能力

- 接真实 LLM。
- 读取当前 slide、选中对象、待办、数据上下文。
- 输出 typed action。
- 写入 revision queue。
- 支持多轮上下文。
- 支持文本选区、图表数据点、表格单元格等更细 target。

### 交互原则

- Agent 修改不能静默发生。
- 涉及结构、图表、数据和风格的修改要记录修订。
- 协作输入天然引用当前选中对象。
- 批注不应是独立留言板，而应是 Agent 修改任务。

## 8.4 微调 liveControl

当前状态：已做第一版。

用户界面统一叫“微调”。内部能力叫 liveControl。

### 页面目标

让用户不用写 prompt，也能安全、直观地调整当前对象。

### 当前控件

整页：

- 设计主色。
- 背景色。
- 圆角。
- 信息密度。
- 动效强度。

标题：

- Typography：Font、Size、Weight、Color、Align、Line、Tracking。
- Size：Width、Height。
- Box：Opacity、Padding、Margin、Border、Radius。
- 换行约束：允许换行 / 保持单行。

正文：

- Typography：Font、Size、Weight、Color、Align、Line、Tracking。
- Size：Width、Height。
- Box：Opacity、Padding、Margin、Border、Radius。
- 换行约束：允许换行 / 保持单行。

图表：

- 图表类型。
- 图表高度。
- 图表数据和结构修改原则上交给 AI，通过修订记录落地。

图片：

- 图片风格。
- 缩放。
- 横向位置。
- Size / Box 基础参数。

### 设计原则

- 控件必须和当前选中对象相关。
- 不同图表或页面的控制状态要隔离，不能互相污染。
- 用户看到的是“设计主色”“背景色”“圆角”，不是 schema 字段。
- 复杂设计标准由 Agent 或 Core 处理，不要求普通用户理解。
- 内容本身有约束时，liveControl 必须暴露约束控制。例如：文本框宽度、是否换行、图表高度。

### 后续控件矩阵

整页：

- 背景。
- 页面密度。
- 页脚显示。
- 标题层级。
- 版式。
- 动效强度。

标题：

- 字号。
- 行高。
- 字重。
- 对齐。
- 文本框宽度。
- 是否允许换行。
- 最大行数。

正文：

- 字号。
- 文本框宽度。
- 段落密度。
- 是否显示要点。
- 是否允许换行。
- 最大行数。
- 语气：管理层、分析型、简报型。

图表：

- 图表类型。
- 指标选择。
- 时间范围。
- 分组维度。
- 是否显示标签。
- 是否显示图例。
- 是否双轴。
- 是否强调拐点。
- 数据点数量。

表格：

- 列选择。
- 排序。
- 小数位。
- 条件高亮。
- 紧凑/宽松。

设计风格：

- 主色。
- 背景。
- 表面色。
- 文字色。
- 边框。
- 圆角。
- 阴影。
- 字体。
- 图表色盘。

## 8.5 批注模型

当前状态：已做第一版。

### 页面目标

让用户像审稿一样指出问题，并把问题提交给 Agent 修改。批注模型服务于 Agent 修订闭环，不是独立评论区。

### 当前能力

- 选择对象：整页、标题、正文、图表。
- 在协作流里提交修改要求。
- 修改要求进入待处理队列。
- 页面触发 `vizual-studio:comment-added` 事件，供内部 Agent 监听。
- Agent 处理后生成修订记录。

### 后续能力

- 文本选区批注。
- 图表数据点批注。
- 表格行列批注。
- 区域框选批注。
- 批注定位高亮。
- 批注回复线程。
- 批注状态：open、in progress、resolved、orphaned。

### 孤立批注定义

当批注原本绑定的对象被删除、改写或无法重新定位时，批注进入孤立状态。

孤立批注不能直接提交修订，必须先重新绑定目标或由 Agent 帮用户迁移到新的对象。

### 设计原则

- 批注必须有 target metadata。
- 批注不能只是纯文本列表，必须能被 Agent 消费。
- 用户看得见批注状态。
- Agent 处理批注后必须生成修订记录。
- 对话和批注可以共用同一个协作流，但批注必须比普通对话多 target、status 和 revision link。

## 8.6 版本与修订

当前状态：已做第一版。

### 页面目标

让 Agent 的修改变成可审阅、可接受、可拒绝的修订，而不是直接覆盖用户内容。

### 当前能力

- 展示完成比例。
- 展示待处理批注数量。
- 展示修订列表。
- 修订可接受。
- 修订可拒绝。

### 后续能力

- 修订 diff。
- 修订前后预览。
- 按 slide 筛选修订。
- 按 Agent 操作筛选修订。
- 回滚到旧版本。
- 从旧对话继续修改。
- 新气泡生成修订结果，而不是覆盖很老的历史消息。

### 设计原则

- Agent 的每一次有意义修改都应有记录。
- 用户可以拒绝 Agent 的修改。
- 老 artifact 应该保留，新的修改应该生成新版本或新消息。

## 8.7 设计风格

当前状态：第一版嵌入在微调里。  
完整设计风格工作台规划中。

### 页面目标

让用户建立适合企业汇报的视觉标准，并让所有 Vizual 内容自动继承。

### 用户语言

界面中应使用：

- 设计风格。
- 主色。
- 背景。
- 字体。
- 圆角。
- 信息密度。
- 动效。

内部可以使用：

- Design.md。
- token。
- theme mapping。
- CSS variables。

但这些不应成为普通用户主路径。

### 标准能力

- 导入标准设计风格。
- 创建设计风格。
- liveControl 调节。
- 应用到 deck。
- 应用到 Vizual 组件。
- 预览所有组件响应。
- 导出标准文件给 Agent/开发者使用。

### 非标准输入

用户可能粘贴混乱的设计描述。Runtime 不负责猜测。

正确流程：

1. 用户给自然语言设计描述。
2. Agent 使用 parser/creator 能力整理成标准设计风格。
3. Runtime 只加载标准结果。
4. Studio 显示映射报告和可调参数。

## 8.8 数据与图表

当前状态：第一版用内置样例数据。

### 页面目标

让 Agent 的数据分析结果在 slide 中可视化，并保留追溯能力。

### 当前能力

- KPI 看板。

## 8.10 内部 Agent 监听模式

## 10. 当前明显缺口

### 10.1 DeckArtifact 还没有下沉到 Vizual Core

现在 deck shell 属于 Studio 层。未来需要在 Vizual Core 中定义 first-class deck artifact，使 Agent 可以稳定生成、修改和测试。

### 10.2 真实 LLM 后端还没有接入

当前 Agent 是前端模拟。内部可以通过 Chrome DevTools 让 Codex/Claude Code 监听，但正式产品需要后端或平台 Agent 接入。

### 10.3 批注定位还不够细

当前只能选整页、标题、正文、图表。后续需要文本选区、图表数据点、表格单元格和自由区域。

### 10.4 设计风格还没有完整工作台

现在只有基础控件。后续需要完整设计风格创建、导入、预览、映射报告和全组件验证。

### 10.5 导出还不够商业级

浏览器打印只是第一步。后续要做高质量 PDF、PNG、HTML bundle 和可选 PPTX adapter。

### 10.6 全部 Vizual 组件尚未在 Studio 内开放

当前只开放 PPT 常用子集。后续需要组件选择器和智能推荐。

### 10.7 项目历史和旧对话续改还没有完整实现

用户三天后打开历史对话并要求修改旧图表，这必须支持。当前只有 localStorage 草稿，未来需要正式 artifact persistence。

## 12. 与 Vizual Core 的关系

Vizual Studio 必须持续推动 Vizual Core 补齐以下能力：

- DeckArtifact schema。
- Block-level target map。
- Comment/revision SDK。
- liveControl schema。
- Export API。
- 全组件主题一致性。
- 设计风格 mapping report。
- Agent action validation。

但 Studio 不应该把所有产品逻辑塞回 Core。Core 保持运行时干净，Studio 负责完整用户体验。

## 13. 与 Claude Design 的关系

Claude Design 给我们的关键启发：

- 对话和持久画布必须同时存在。
- 用户需要直接编辑，不是一切都靠 prompt。
- 批注必须绑定具体对象。
- 自定义调节器比纯自然语言更稳定。
- 设计系统应该成为项目上下文，而不是每次重新描述。
- 导出和 handoff 是产品闭环，不是附属功能。

Vizual Studio 不复制 Claude Design 的泛设计定位，而是把这些交互原则应用到：

- 数据分析。
- 经营汇报。
- 商业 PPT。
- Agent 可视化输出。

## 14. 与 Hyperframes 的关系

Hyperframes 暗示 HTML 内容可以继续走向动画、视频和更高级的视觉叙事。

Studio 第一版不集成 Hyperframes，但要保留方向：

- HTML deck 可加入 motion tokens。
- 未来可生成动态汇报。
- 未来可导出视频或动效演示。
- Agent 可通过 CLI 生成和验证复杂视觉序列。

## 17. 一句话总结

Vizual Studio 要成为“AI Agent 的商业演示工作台”。用户不需要懂前端、不需要懂 schema、不需要懂设计标准文件；他们只需要和 Agent 一起在一个专业画布里看、改、批注、微调和交付。Vizual Core 提供运行时，Studio 提供完整产品体验。
